# Issue: #8 — COLOR_POOL を SPEC準拠の12色に
Stage: 1/5 Architect

## 方針（1行）
`src/domain/colors.ts` の `COLOR_POOL` を、既存14候補から色相＋知覚輝度＋色覚多様性で選んだ12色の純データ配列に確定し（コメントアウト廃止）、配布アルゴリズムは不変のままテスト/メッセージ/参照箇所を12色前提に整合させる。

## 設計方針（アーキテクチャ / データフロー / IF / DB / エラー）
- データ層のみの変更。`COLOR_POOL: readonly AssignedColor[]` を**全要素有効な12要素配列**に書き換え、コメントアウト方式（「戻すときはコメントを外す」運用）を撤廃。`{ hex, name }` ペア構造と `as const` は維持。
- `MAX_MEMBERS = COLOR_POOL.length` の導出式は不変 → 自動的に 12 になる。`assign-colors.ts` の `assignColorsToTrip` / `pickColorForJoiner` / `availableColors` / `usedColorHexes` のロジックは一切触らない（配布順・シャッフル・残り色抽出はプール内容に非依存）。
- `contrastTextColor` も不変。配列中の各 hex がこの関数（閾値 L>150 で黒文字）と整合するか採用時に検証済み（後述）。`しろ#FFFFFF` は L=255 で黒文字、`ColorChip` が枠線付き描画するため**残す**。
- DB変更なし。Firestore セキュリティルール（Issue #6）は既に上限12を強制済みで本PRでは触らない。アプリ側を12に揃えるだけで両者が一致する。
- エラーハンドリング不変。`TooManyMembersError` は `max = MAX_MEMBERS` を既定引数で受けるため、メッセージは自動で「limit of 12」になる。ハードコードされた数値「8」は存在しない（要 Investigator 確認）。

## 選定根拠（色覚多様性が核心 / SPEC §6）
- **近接3群の取捨を「明度差」で解く**: 3つの近接群はいずれも色相が混同しやすいので、群内では知覚輝度を大きく離す。
  - 黄系（きいろ216 / やまぶき170 / だいだい140）: 両端の **きいろ(216) と だいだい(140) を採用**し、中間の **やまぶき(170)を脱落**。やまぶきは赤緑色覚（P/D型）でだいだい・きいろ両方と混同しやすく、明度も中間で差別化が弱い。
  - 緑系（きみどり182 / みどり121 / あおみどり129）: **3色とも採用**。きみどり(182)は明度が突出して高く、みどり(121)とあおみどり(129)は色相が黄寄り/青寄りで分離。みどり/あおみどりは明度が近い(121/129)が、隣接配置を避ければ実用上分離可能。SPEC §6 の「名前ラベル必須」が最終的な保険。
  - 青系（あお108 / みずいろ170 / あいいろ73）: **3色とも採用**。みずいろ(170・明)/あお(108・中)/あいいろ(73・暗)と明度が階段状に分かれ、色覚多様性下でも明度だけで判別可能。
- **赤緑混同への配慮**: あか(99)・みどり(121) は P/D型で混同しうるが、明度差＋名前ラベルで担保。さらに あか/だいだい/ちゃいろ(赤橙茶) は L=99/140/103 と離し、ちゃいろは彩度を落とした茶で赤橙と分離。
- **青黄混同（T型）への配慮**: あお系ときいろ・だいだいは明度を大きく離してあり、T型でも明度コントラストで判別可能。
- **しろ#FFFFFF を落とす判断**: L=255 で `きいろ(216)` と並ぶ最高明度域に2色を割くのは色覚多様性上もったいなく、白は背景/カードと同化しやすく `ColorChip` 側で常時枠線を要する例外色。**12枠を有彩色＋茶で埋めるため しろを脱落**させ、明度・色相の分布を最大化する。代わりに藍(あいいろ73)が最暗を担う。
  - 補足: `contrastTextColor` 自体は しろ を落としても影響なし（汎用関数）。`profile.tsx` / `trip/[id]` が呼ぶだけ。

## 採用理由とトレードオフ
- **採用: しろを落として有彩色12色** — 明度・色相の分散が最大化され色覚多様性要件を最もよく満たす。トレードオフ: 既存 seed/UI で しろ を名前参照していると壊れる（→ seed は あお/あか/みどり/きいろ/もも のみ参照で しろ 不使用、安全と確認済み）。
- **却下案A: しろを残しやまぶきとあおみどりを落とす8→12維持寄り** — 既存有効8色の体裁を保てるが、白が高明度域を浪費し近接色分離が弱い。色覚多様性要件を満たしにくいため却下。
- **却下案B: 14候補すべてを採用しプールを14に拡張** — 選定不要で楽だが、上限12（セキュリティルール）と不整合になり、近接色2ペアが混同源として残る。SPEC違反のため却下。

## テスト更新方針
- **全人数配布**: `makeTrip(MAX_MEMBERS)`（=12）で全員に色が付き、hex集合サイズ=12（重複なし）。
- **上限**: `makeTrip(MAX_MEMBERS+1)`（=13）で `TooManyMembersError`、12人はOK（既存 `ちょうど12人なら配布できる` がそのまま機能）。
- **途中参加**: 4人配布済み → `availableColors` が `MAX_MEMBERS - 4`（=8）件、`pickColorForJoiner` が未使用色を返す。満員（12色使用済み）で `TripIsFullError`。
- **順序非依存**: `COLOR_POOL[0/1/2/4]` 参照テストは自己同値比較なのでプール確定後もそのまま通る。期待値ハードコードは追加しない。
- ゲート: `npx tsc --noEmit` 0件、`npx jest` 全 pass。
